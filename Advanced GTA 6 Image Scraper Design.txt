Architectural Blueprint for a Cognitive Visual Asset Curation Sidecar: Implementing Groq-Powered Semantic Filtering and Zero-Download
High-Resolution Validation for Leonida Vice
The digital asset architecture of the Leonida Vice platform is governed by a strict,
luxury-cinematic dark-mode aesthetic that treats every publication as a premium film poster. Under the strict platform guidelines outlined in ARTICLE_LAYOUT_RULES.md, every published article must feature exactly two unique, high-definition (HD) visual assets scraped directly from the original source website: a full-bleed hero header image and a distinct body image. If the ingestion pipeline fails to locate both unique, high-quality images directly from the source page, the article is discarded and excluded from publication to prevent rendering empty arrays, distorted graphics, or generic fallback stock photography.
To automate this strict curation process without introducing latency or layout degradation, the proposed solution establishes an asynchronous, supervisor-style visual asset pipeline: the Cognitive Image Scraping and Curation Sidecar (CISCS). Designed to run as a separate service alongside the current article text scraper, the CISCS leverages the Model Context Protocol (MCP) and a decoupled sidecar pattern to intercept crawled article text, execute semantic relevancy checks via the built-in Groq API, enforce Grand Theft Auto 6 (GTA 6) contextual filters, and validate strict 2K resolution standards.

Decoupled Architectural Design of the Visual Supervision System
The visual curation sidecar operates on an asynchronous, event-driven pattern designed to isolate resource-heavy computer vision tasks and network-intensive image extraction from the primary text scraper. In synchronous scraping architectures, executing deep text classification, third-party search queries, and high-resolution image decoding within a single execution thread introduces severe processing bottlenecks and increases Interaction to Next Paint (INP) latency. By deploying the visual scraper as an independent sidecar service, the system achieves a robust separation of concerns, keeping the primary publishing pipeline responsive and isolated from downstream network failures.
Communication between the primary article scraper and the visual sidecar is orchestrated via a non-blocking Redis message broker. When the article scraper discovers and parses a raw feed, it commits the basic text payload to the database and pushes an ingest job to a Redis list (visual_supervisor:jobs) containing the article slug, raw scraped HTML, and any initial media URLs extracted from the page. To prevent generic Grand Theft Auto (such as legacy GTA 5 or Grand Theft Auto Online) or older franchise graphics from polluting the Leonida Vice database, the visual supervisor sidecar utilizes the built-in Groq API to run real-time semantic analyses of the scraped article text.
Simple keyword regex matching is highly fragile and fails to capture contextual nuance, often leading to false positives where legacy assets are mistakenly classified as relevant. By querying the llama-3.3-70b-versatile model, the CISCS evaluates the central semantic theme of the text. To guarantee deterministic, programmatic behavior, the system utilizes the Groq Structured Outputs feature with strict schema adherence (strict: true). This ensures that the model is constrained at the token level to only generate valid JSON that strictly conforms to a defined Pydantic schema, eliminating conversational formatting anomalies and the need for complex regex post-parsing. Web scraping at scale presents significant network overhead and processing latency when downloading multi-megabyte 2K and 4K images just to verify their pixel dimensions.
Downloading entire raster payloads that are ultimately discarded for failing resolution checks introduces prohibitive latencies and storage bottlenecks. To bypass this overhead, the CISCS implements a zero-download dimensional filtering pipeline.
The visual supervisor uses Python's requests library in streaming mode (stream=True) combined with Pillow's ImageFile.Parser to decode the image dimensions incrementally from the initial byte headers of the file, terminating the connection immediately once the dimensions are read. Because web servers write metadata headers (such as the PNG IHDR chunk or JPEG SOF0 marker) at the absolute beginning of the file, the width and height are contained within the first few hundred bytes.

Perceptual Hashing and Visual Deduplication Engines
The ARTICLE_LAYOUT_RULES.md document mandates that the Hero and Body images must be completely unique and different from one another. A critical failure mode in web scrapers is the duplicate ingestion of the exact same graphic under different filenames, or the sourcing of near-identical frames (e.g., frames extracted millisecond-apart from the same trailer).
To prevent visual duplication, the CISCS implements a robust visual hashing engine using the Python imagehash library. The sidecar generates Perceptual Hashes (pHash) and Difference Hashes (dHash) for every candidate image. While cryptographic hashing (like SHA-256) yields completely different strings for a single-pixel modification, perceptual hashing produces mathematically close strings for visually similar inputs.
The Difference Hashing (dHash) algorithm is selected for its high computation speed and low processing latency. The pipeline executes the following operational steps to calculate the dHash:
1. Grayscale Conversion: Eliminates color channels to analyze structural luminance.
2. Downsizing: Resizes the image to a (n+1) \times n matrix (typically 9 \times 8).

3. Luminance Difference Calculation: Compares adjacent horizontal pixels.
4. Binary Construction: Generates a 64-bit or 128-bit hexadecimal fingerprint string.

Because MongoDB does not natively support Hamming distance calculations efficiently via B-tree indexes, storing hashes solely as raw hex strings can lead to expensive table scans
during bulk deduplication lookups. To optimize performance at scale, the database architecture decomposes the 64-bit dHash strings into four 16-bit integer segments (hash_segment_1, hash_segment_2, etc.) and indexes them individually.
This segment decomposition allows the supervisor to execute quick, exact-match queries against individual segments, filtering out 99% of non-matching records in milliseconds before running the precise bitwise XOR Hamming calculation in memory on the remaining candidates. This multi-layered validation pattern guarantees that all active news releases comply with the platform's rigorous design guidelines, protecting the site's layout integrity and ensuring a flawless, highly polished cinematic user experience.
